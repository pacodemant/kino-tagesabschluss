## Conventions

- **Base URL:** `https://<your-host>` (default port `8080`).
- **Content type:** `application/json` for all request and response bodies.
- **Money is always integer cents.** Every monetary amount in requests and
  responses is an integer number of cents, e.g. `123456` means `1234.56 EUR`.
  Never send floats/decimals.
- **Dates** are `YYYY-MM-DD` (e.g. `2026-06-10`).

## Typical flow

1. **Ensure a daily report exists** for the location and date, and obtain its
   `report_id` → `POST /api/daily-reports/ensure`.
2. **Submit the settlements** (cash totals + terminal closings) for that
   `report_id` → `PUT /api/daily-reports/{id}/settlements`.

Das Buchhaltungssystem des Kinos heißt Flurbocash und hat eine REST-API — eine strukturierte Schnittstelle, über die die App Daten in einem genau vorgegebenen Format übermittelt. Statt eines einfachen Webhooks wie bisher sind künftig zwei Aufrufe pro Abrechnung nötig.

Was sich gegenüber heute ändert: Bisher ein einfacher POST mit Formulardaten. Künftig zwei aufeinanderfolgende Aufrufe mit JSON — und eine genau definierte Datenstruktur für EC-Kartendaten.

## `POST /api/daily-reports/ensure`

Returns the daily report ID for a location and date, **creating the report if it
does not exist yet**. Idempotent: calling it again for the same location+date
returns the same `report_id`.

## `PUT /api/daily-reports/{id}/settlements`

Submits the cash totals and per-terminal card closings for a daily report.
`{id}` is the `report_id` from the ensure call.
1
Formular anlegen oder abrufen — POST /api/daily-reports/ensure App schickt Standort-ID (location_id) und Datum. Flurbocash antwortet mit einer report_id. Existiert für diesen Tag bereits ein Bericht, wird einfach die vorhandene ID zurückgegeben — kein Fehler, kein Überschreiben. Idempotent: beliebig oft aufrufbar, immer dasselbe Ergebnis.
2
Daten übermitteln — PUT /api/daily-reports/{id}/settlements App schickt die gezählten Barbeträge und die EC-Aufschlüsselung pro Terminal an die report_id aus Schritt 1. Flurbocash antwortet mit dem berechneten Differenzbetrag.
Was Flurbocash selbst kennt: Den Soll-Wert aus dem Kassensystem. Die App liefert nur den Ist-Wert (gezähltes Bargeld + EC). Den Vergleich zieht Flurbocash intern.

TID-Absicherung — Whitelist-Konzept

Die Terminal-ID (TID) wird vom BelegScan aus dem Foto gelesen. Eine falsch erkannte TID (z. B. 0 statt 8) führt zu einem 400-Fehler von Flurbocash — die Daten landen nirgends. Zusätzliches Risiko: Flurbocash prüft ob die TID für den Standort registriert ist.

  • Whitelist pro Standort in der Konfiguration Jedes Standort-Gerät kennt die für diesen Standort gültigen TIDs. Nach dem Scan prüft die App: ist die gelesene TID in der Liste? Falls nicht → Warnung vor dem Prüf-Popup, MA muss explizit bestätigen oder korrigieren.
    Geplant
  • TID im Prüf-Popup immer sichtbar und editierbar Die erkannte TID wird im Popup fett und prominent angezeigt. MA kann sie direkt korrigieren bevor er bestätigt — auch ohne Whitelist-Treffer.
    Geplant
  • TIDs liefert die IT einmalig beim Setup Die Kino-IT kennt alle registrierten TIDs in Flurbocash. Sie trägt sie einmalig in die Gerätekonfiguration ein — danach läuft die Prüfung automatisch.
    Offen — IT

## Corrections

To correct a settlement you already submitted, send it again **with its
`settlement_number`** (1–4). This overwrites that settlement in place.

{ "settlements": [{ "settlement_number": 2, "cash_total": 98765,
                    "terminals": [{ "tid": "TID-ALPHA", "girocard": 9000 }] }] }

Rules:
- The targeted `settlement_number` must already exist. Targeting a non-existent
  number returns `400`.
- Corrections are only possible while the report is **not finalized**.
- A day allows at most **4** settlements.

Solange ein Bericht in Flurbocash nicht finalisiert ist (das macht die Buchhaltung, nicht der MA), kann er überschrieben werden. Dafür wird Schritt 2 erneut gesendet — diesmal mit "settlement_number": 1.

  • Erster Versand (normal) Kein settlement_number — Flurbocash legt den Eintrag neu an und vergibt die Nummer automatisch (1, 2, …).
  • Korrektur "settlement_number": 1 mitschicken — Flurbocash überschreibt den Eintrag. Die Ziel-Nummer muss bereits existieren, sonst Fehler.
  • Konsequenz für die App Der "Erneut senden"-Button im Verlauf sendet künftig einen Korrektur-Call mit settlement_number: 1 — nicht einen neuen Eintrag. Voraussetzung: die ursprüngliche report_id muss lokal gespeichert bleiben.
Maximale Abrechnungen pro Tag: 4. Mehr Aufrufe werden von Flurbocash mit 400 abgelehnt. Für alle Standorte außer Bar Tabak (2 Abrechnungen/Tag) kein Problem.
Fehlermeldungen — was die App daraus macht
Original · EXTERNAL_API_Schauburg.md → Errors
| Status | Meaning                                                            |
|--------|-------------------------------------------------------------------|
| `400`  | Malformed body, missing/unknown `tid`, bad date, `settlement_number` outside 1–4 |
| `400`  | More than 4 settlements (`maximum of 4 settlements per day reached`) |
| `400`  | `settlement_number` targets a settlement that does not exist       |
| `400`  | Report is finalized and can no longer be modified                 |
| `401`  | Missing or invalid/revoked API key (`X-API-Key`)                  |
| `403`  | API key is not authorized for the requested location              |
| `404`  | Daily report `{id}` not found                                     |
| `500`  | Server error                                                      |

On a `400`, no data is written — the request is validated before any changes are
persisted, so a rejected call leaves the report unchanged.

Flurbocash antwortet bei Fehlern mit HTTP-Statuscodes. Die App muss diese abfangen und dem MA eine verständliche Meldung zeigen — keine technischen Codes, keine englischen Fehlertexte.

  • 400 — Falsches Format, unbekannte TID, Bericht finalisiert „Übertragung fehlgeschlagen. Prüfe TID und Werte, dann erneut senden." — bei finalisierten Berichten: „Diese Abrechnung wurde bereits abgeschlossen."
    400
  • 401 — API-Key fehlt oder ungültig „Zugang nicht autorisiert — bitte Einstellungen prüfen oder IT kontaktieren."
    401
  • 403 — API-Key für diesen Standort nicht berechtigt „Dieser API-Key ist für den gewählten Standort nicht gültig — bitte IT kontaktieren."
    403
  • 404 — Bericht-ID nicht gefunden Intern abfangen — sollte nach erfolgreichem Schritt 1 nicht auftreten. Falls doch: erneut ab Schritt 1 starten.
    404
  • 500 — Server-Fehler „Server nicht erreichbar — bitte später erneut versuchen."
    500

## Authentication

All external endpoints are authenticated with an **API key**, sent in the
`X-API-Key` header:

    X-API-Key: fck_2a843e19...

Each API key is **scoped to one or more locations**. A key may only read/write
data for the locations it is assigned to. Requests for any other location are
rejected with `403 Forbidden`.
  • ApiUploadService: 1 Call → 2 Calls Aus dem bisherigen einzelnen POST werden zwei aufeinanderfolgende Aufrufe. Die report_id aus Schritt 1 fließt in Schritt 2 — sie muss auch lokal für spätere Korrekturen gespeichert werden.
    Run
  • Format: form-encoded → JSON Content-Type wechselt von application/x-www-form-urlencoded auf application/json. Felder werden als JSON-Objekt strukturiert statt als Schlüssel-Wert-Paare gesendet.
    Run
  • location_id ins Kino-Modell Jedes Kino-Objekt in kino.dart bekommt ein neues Feld locationId (int). Wird mit den Werten der IT befüllt.
    Wartet auf IT
  • report_id lokal speichern Nach Schritt 1 muss die report_id im lokalen Speicher abgelegt werden — damit ein Korrektur-Call aus dem Verlauf heraus möglich ist ohne erneuten Schritt-1-Aufruf.
    Run
  • Konfigurationsbereich: PIN-Schutz Der Dev-/Verwaltungsbereich in den Einstellungen wird mit einem vierstelligen PIN geschützt. location_id, API-Key, Basis-URL und TID-Whitelist werden dort einmalig beim Geräte-Setup eingetragen.
    Run
  • "Erneut senden"-Logik anpassen Statt eines neuen Eintrags sendet der Button einen Korrektur-Call mit settlement_number: 1. Voraussetzung: lokal gespeicherte report_id.
    Run
  • Fehlerbehandlung im Upload-Service Alle definierten HTTP-Statuscodes (400, 401, 403, 404, 500) werden abgefangen und in verständliche deutsche Meldungen für den MA übersetzt.
    Run

terminals[].girocard    | int | EC-Karte: Girocard
terminals[].lastschrift | int | EC-Karte: Lastschrift
terminals[].mastercard  | int | Kreditkarte: Mastercard
terminals[].visa        | int | Kreditkarte: Visa
terminals[].maestro     | int | Kreditkarte: Maestro
terminals[].vpay        | int | Kreditkarte: V-Pay

Die vom BelegScan gelesenen Kartentypen entsprechen fast 1:1 den Flurbocash-Feldern. Das Mapping ist eindeutig:

  • BelegScan: "girocard"→ Flurbocash: girocard
  • BelegScan: "SEPA Lastschrift"→ Flurbocash: lastschrift
  • BelegScan: "MasterCard"→ Flurbocash: mastercard
  • BelegScan: "VISA"→ Flurbocash: visa
  • BelegScan: ggf. "Maestro"→ Flurbocash: maestro
  • BelegScan: ggf. "V Pay"→ Flurbocash: vpay
Hinweis: Der BelegScan-Prompt fasst Untervarianten bereits zusammen ("girocard Online" + "girocard PIN" → "girocard"). Das Mapping-Ergebnis ist daher direkt verwendbar — kein zusätzlicher Normalisierungsschritt nötig.

Bewusste Entscheidungen — was wir nicht tun
Kein Variablen-Refactoring
Interne App-Variablen (z. B. bargeldGesamt) werden nicht auf API-Feldnamen umbenannt. Das Mapping von intern → API-Format geschieht ausschließlich im ApiUploadService. Refactoring vor abgeschlossener Integration wäre riskant und unnötig.
Lokale Datenspeicherung bleibt erhalten
Die API ersetzt nicht die lokale Datenhaltung. Verlauf, Entwürfe und die gespeicherte report_id für Korrektur-Calls bleiben in SharedPreferences / Hive. Die API ist der Übertragungskanal am Ende des Workflows — nicht dessen Ersatz.
Konfiguration in SharedPreferences, nicht in Dateien
Der Web Build ist ein statisches Paket — es gibt keine beschreibbaren Konfigurationsdateien auf dem Server. Alle editierbaren Werte (API-Key, location_id, TIDs, E-Mail) liegen im Browser-Speicher des jeweiligen Geräts und werden über den PIN-geschützten Einstellungsbereich gepflegt.
Kein Remote-Konfigurationsdashboard in V1
Ein zentrales Dashboard zum Verwalten aller Geräte aus der Ferne wäre ein eigenständiges zweites System (eigener Server, Datenbank, Auth). Bei fünf Standorten und selten wechselnden Werten steht der Aufwand nicht im Verhältnis. Einmalig manuelles Setup pro Gerät reicht. → Nice-to-have, Roadmap nach V1.
Neues Feature — Stapel-Scanner für EC-Belege

Hintergrund: Die zuständige Buchhaltungsperson ist mehrere Wochen krankgeschrieben. Aufgestaute EC-Belege müssen nacherfasst werden. Das Feature stärkt außerdem die Verhandlungsposition — es adressiert einen konkreten, akuten Bedarf.
