Technische Übergabedokumentation

Dieses Dokument richtet sich an die Kino-IT. Es beschreibt Aufbau, Konfiguration, Deployment und Wartung der App vollständig genug, um ohne Rückfragen einsteigen zu können.

Entwicklungsumgebung einrichten
1
Flutter installieren Flutter SDK von flutter.dev herunterladen. Version: SDK ^3.8.0 oder neuer. Nach Installation: flutter doctor ausführen — alle Checkmarks sollten grün sein (Chrome reicht, iOS/Android nicht zwingend nötig).
2
Repository klonen git clone [repo-url] — Branch master ist der Hauptbranch. Kein Develop-Branch.
3
Abhängigkeiten installieren flutter pub get im Projektverzeichnis ausführen.
4
App lokal starten flutter run -d chrome — öffnet die App direkt im Browser. Änderungen im Code werden mit r (Hot Reload) oder R (Hot Restart) sofort sichtbar.
5
Codequalität prüfen flutter analyze — zeigt Warnungen und Fehler. Keine offenen Warnungen vor einem Commit.
Kein manueller Build nötig: GitHub Actions baut und deployt automatisch bei jedem Push auf master. Lokale Builds nur für Tests.
Technischer Stack
Framework
Flutter (Dart) — kompiliert zu einer statischen Web-App. Kein Backend, keine Datenbank serverseitig. Kein App-Store.
Deployment
GitHub Actions — bei jedem Push auf master automatisch Build + Deploy. Konfiguration unter .github/workflows/.
Datenpersistenz
SharedPreferences = localStorage im Browser. Daten bleiben gerätespezifisch erhalten bis der Browser-Cache geleert wird. Wird vom Cache geleert: nur lokaler Verlauf, nicht die bereits exportierten Abrechnungen.
Datenexport
HTTP POST nach Abschluss an konfigurierbare URL. Format: application/x-www-form-urlencoded (kein CORS-Preflight). URL und API-Key in den App-Einstellungen (Dev-Bereich).
Konfigurationsdateien
Getränkelisten und Wechselgeldbestand als Textdateien im Repo unter config/. App lädt sie beim Start und cached sie lokal in SharedPreferences.
Service Worker
Bewusst deaktiviert. Jeder App-Aufruf lädt die aktuelle Version frisch vom Server — kein veralteter Cache auf Mitarbeiter-Geräten.

Datenfluss einer Abrechnung
1
Schritt 1 — Bargeld zählen Mitarbeiter zählt Scheine, Münzrollen und lose Münzen. Zwischenspeicherung nach jeder Eingabe in SharedPreferences (Key: abrechnung_[kinoId]_[datum]). Wechselgeldkontrolle läuft parallel.
2
Schritt 2 — Belege & Einnahmen Kino-Soll, Bistro-Soll, Ausgaben und EC-Belege werden eingegeben. Differenz Anfangsbestand. Entwurf ebenfalls in SharedPreferences gespeichert.
3
Schritt 3 — Abschluss Alle Werte aus Schritt 1 und 2 werden über TagesabschlussFinalisierenUsecase zu einem TagesabschlussFinal-Objekt zusammengeführt. Lokal in SharedPreferences gespeichert. HTTP POST an konfigurierte URL gesendet.
4
Export ApiUploadService.upload() sendet alle Felder als Form-POST. Bei CORS-Fehler: Upload gilt als gesendet, Empfang nicht bestätigbar — kein Doppel-Upload. Erneut senden möglich aus dem Verlauf heraus.

Wichtige technische Details
€
Geldbeträge intern in Cent Alle Beträge werden als int in Cent gespeichert und verarbeitet — keine Gleitkomma-Rundungsfehler. Konvention: Variablenname endet auf Cent (z. B. kinoSollCent). Anzeige und Export in Euro.
⏰
Logischer Abrechnungstag (6-Uhr-Knick) 0:00–5:59 Uhr = Vortag. Ab 6:00 Uhr = heute. Implementiert in DatumsHelper.logischerAbrechnungsTag(). Steuert alle Datums-Zuordnungen.
🔑
Kino-IDs sind stabile Schlüssel Die IDs kino_01 bis kino_05 dürfen nie geändert werden — sie sind Teil der SharedPreferences-Keys. Eine Änderung würde alle gespeicherten Daten auf den Geräten der Mitarbeiter unzugänglich machen.
📋
Bekannte offene Punkte (Stand Juni 2026) Gondel (kino_02) und Bar Tabak (kino_05) sind noch nicht vollständig implementiert. Kein globaler State-Management-Layer (Provider) — jede Seite lädt ihren Zustand selbst. Geplant für einen späteren Refactoring-Run.
Häufige Probleme
Export schlägt fehl
Häufig CORS. Der POST kommt am Server an, der Browser liest die Antwort nicht. Lösung: Server muss Header Access-Control-Allow-Origin: * senden. Aus dem Verlauf kann ein fehlgeschlagener Export manuell wiederholt werden.
Getränkeliste leer
Config-Datei beim Start nicht erreichbar (kein Netz). Einstellungen → Getränkeliste → „Standard-Liste herunterladen" ausführen.
Falscher Wechselgeldbestand
config/wechselgeld.txt aktualisieren und pushen. Mitarbeiter beim nächsten App-Start informieren — der Wert aktualisiert sich automatisch.
Daten nach Browser-Reset weg
SharedPreferences = localStorage. Wird beim Löschen des Browser-Caches geleert. Bereits exportierte Abrechnungen sind beim Kino-Server — nur der lokale Verlauf geht verloren.
App zeigt alte Version
Service Worker ist deaktiviert, sollte nicht auftreten. Harter Reload: Strg+Shift+R (Windows) / Cmd+Shift+R (Mac).
Dev-Modus aktivieren
Einstellungen → ganz nach unten scrollen → Dev-Kachel antippen → Entwicklermodus einschalten. Zeigt API-URL/Key-Felder, Testwerte-Button und weitere Debug-Funktionen.
