Konzept: EC-Beleg-Scan

Automatische Erfassung und Aufschlüsselung von EC-Kartenumsätzen per Kamerascan. Pilot: Schauburg. Ziel: V1.

Ausgangslage & Mehrwert

Heute notiert der Mitarbeiter beim Abschluss nur den Gesamtbetrag des EC-Umsatzes — abgelesen vom Kassenschnitt-Ausdruck des Terminals. Die Buchhaltung muss anschließend jeden einzelnen EC-Beleg manuell nach Kartenart aufschlüsseln: girocard, SEPA Lastschrift, MasterCard, VISA. Das kostet Zeit und ist fehleranfällig.

Heute: Mitarbeiter liest ab 1 Zahl, manuell Heute: Buchhaltung schlüsselt auf je Beleg, manuell
Künftig: Mitarbeiter fotografiert Bon 1 Tap Künftig: App liefert Aufschlüsselung automatisch, strukturiert Künftig: Buchhaltung erhält pro Abschluss Terminal-ID, Kartenart, Anz., Betrag
Aufwand Buchhaltung entfällt
Datenfluss
1
Mitarbeiter tippt „Scan" in Schritt 2 Kamera-Button neben dem EC-Betragsfeld. Kamera öffnet sich via image_picker.
2
Foto wird aufgenommen Mitarbeiter fotografiert den Kassenschnitt-Ausdruck des EC-Terminals.
3
Bild + Prompt → Anthropic API App kodiert das Bild als base64. HTTP POST an Anthropic API mit festem System-Prompt und API-Key aus den Einstellungen. Kein eigenes Backend nötig.
4
Anthropic antwortet mit JSON Strukturierte Daten: Terminal-ID, Datum, Kartenarten mit Anzahl und Betrag in Cent, Gesamtsaldo.
5
Felder werden automatisch befüllt EC-Gesamtbetrag, Aufschlüsselung nach Kartenart, Terminal-ID. Mitarbeiter prüft kurz, bestätigt.
6
Export wie bisher — mit neuen Feldern Kartenarten-Aufschlüsselung geht zusätzlich in den HTTP POST an die Buchhaltungs-API.
System-Prompt (fest in der App)

Der Prompt ist eine Konstante im BelegScanService — nicht vom Nutzer editierbar, nicht in einer Config-Datei. Anpassungen nur durch einen CC-Run.

Du bist ein Belegscanner für Kassensysteme. Du bekommst ein Foto eines EC-Kartenbelegs (Kassenschnitt). Extrahiere folgende Felder und antworte NUR mit JSON, ohne Erklärung, ohne Markdown-Backticks: { "terminal_id": "...", "datum": "TT.MM.JJJJ", "uhrzeit": "HH:MM", "zahlungsarten": [ { "art": "girocard", "anzahl": 2, "betrag_cent": 7920 }, { "art": "SEPA Lastschrift", "anzahl": 4, "betrag_cent": 6890 }, { "art": "MasterCard", "anzahl": 1, "betrag_cent": 2600 }, { "art": "VISA", "anzahl": 1, "betrag_cent": 3720 } ], "gesamt_anzahl": 8, "gesamt_betrag_cent": 21130 } Beträge immer in Cent (ganzzahlig, kein Komma). Wenn ein Feld nicht lesbar ist: null. Nur die Zahlungsarten ausgeben die auf dem Beleg stehen.
Warum Cent: Die App rechnet intern ausschließlich in Cent (kein Float, keine Rundungsfehler). Das JSON-Format ist darauf abgestimmt — kein Umrechnungsschritt nötig.

API-Key — Verantwortung & Setup
Wer erstellt den Key
Die Kino-IT legt ein Konto auf console.anthropic.com an, hinterlegt eine Kreditkarte und generiert einen API-Key. Eigener Account, eigene Kostenkontrolle.
Wo liegt der Key
Nur in den App-Einstellungen (Dev-Modus) — in SharedPreferences des jeweiligen Geräts. Nie im Quellcode, nie im Repository.
Einmalig pro Gerät
Jedes Gerät auf dem die App läuft muss den Key einmal eingetragen bekommen. Danach persistent gespeichert.
Kosten
Ca. 1.000–2.000 Token pro Scan (Bild + JSON-Antwort). Bei Sonnet: Bruchteile eines Cents. Bei 180 Abrechnungen/Monat unter 1 €/Monat.
Sicherheit
Der Key liegt im Browser-LocalStorage — sicherer als im Code, aber nicht produktions-sicher gegen einen entschlossenen Angreifer mit Gerätezugang. Für den internen Einsatz auf Mitarbeitergeräten akzeptables Risiko. Bei höherem Sicherheitsbedarf: Proxy-Server (Cloudflare Worker o. ä.) als nächste Stufe.

Offene Fragen vor Umsetzung
Export-Format Kartenarten
Wie sollen die Zahlungsarten im HTTP POST an die Buchhaltungs-API übergeben werden? Flache Keys (ec_girocard_betrag_cent) oder JSON-Array? Abhängig vom Format das die Kino-IT vorgibt.
Fallback bei schlechtem Foto
Wenn der Scan ein Feld nicht lesen kann (null), bleiben die betroffenen Felder leer und werden rot hervorgehoben — Mitarbeiter trägt manuell ein. Kein harter Fehler, kein Scan-Zwang.
Offline-Verhalten
Kein Netz = kein Scan. Button wird ausgegraut, Felder bleiben manuell eingebbar. Kein Datenverlust, kein Workflow-Stopp.
