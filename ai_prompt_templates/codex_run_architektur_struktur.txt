# codex_run_architektur_struktur.txt
# Zweck:
# - Template für Struktur-/Architekturarbeiten (z.B. Usecase-Layer, Entkoppelung UI/Storage, Ordnerstruktur innerhalb lib/).
# - Wird genutzt, wenn wir bewusst Schichten trennen oder technische Schuld reduzieren.
# - Bestimmt: enge Grenzen, damit nichts „nebenbei“ umgebaut wird.

Codex Run <NUMMER>: <Kurzbeschreibung Architektur/Struktur>

Kontext:
- Warum ist diese Strukturänderung nötig? (1–3 Sätze)
- Betroffene Dateien/Ordner: <Liste>
- Was ist aktuell problematisch? <kurz>

Ziel:
- Was wird ausgelagert / neu eingeführt / getrennt?
- Was bleibt unverändert (UI, Keys, Modelle)?

Verbindliche Regeln:
1) Keine UI-Änderungen (nur Verkabelung/Calls).
2) Keine Key-/JSON-Änderungen.
3) Keine fachliche Logik neu erfinden; nur verschieben/kapseln.
4) Neue Dateien nur, wenn unbedingt nötig – und dann minimal.

Technische Vorgaben:
- Clean-Arch-ähnlich, aber pragmatisch: Usecase/Service als dünne Schicht zwischen UI und Storage.
- Keine neuen Packages.
- Bestehende Benennungen beibehalten.

Erwartetes Ergebnis:
- Welche Schicht bekommt welche Verantwortung?
- Liste der geänderten/neu angelegten Dateien.

Abschluss:
- flutter analyze muss sauber sein.
- flutter test muss grün sein (falls Tests vorhanden).
- Liste der geänderten Dateien.
